# ЦИФРОВОЙ ПАСПОРТ И СТАНДАРТ РАЗРАБОТКИ CRM (EXECUTABLE SPECIFICATION)

## 📊 Краткий цифровой паспорт архитектуры
1. **crm_nginx_gateway (Port 80)**: Точка входа. `/api/v1/logs/` -> `logger:8001`, `/api/v1/analytics/` -> `analyzer:8002`, `/api/` -> `backend:8000`.
2. **crm_backend_core (FastAPI, Port 8000)**: Главное ядро, работа с PostgreSQL через SQLAlchemy + asyncpg.
3. **crm_qa_orchestrator (FastAPI, Port 8005)**: Контролирующий орган. Перед каждым сценарием делает `TRUNCATE ... CASCADE` в БД, симулирует фронтенд на BS4 и бьет по API через Gateway.
4. **crm_logger_service (FastAPI, Port 8001)**: Асинхронный логгер таймлайна (коды 0000-0601).
5. **Матрица статусов**: 
   * Логистический: `CANDIDATE` ➡️ `IN_REQUEST` ➡️ `IN_DELIVERY` ➡️ `RECEIVED`.
   * Физический: `IN_STORE` (на полке), `SOLD` (чеком), `LOST/WRITE_OFF` (брак), `IN_DISASSEMBLED` (разукомплектован), `ABSORBED` (собран).

## 📐 Регламент Top-Down Contract-Driven (Разработка сверху вниз)
1. **Шаг А**: Создается текстовый `.feature` сценарий на русском с мета-тегами `@Trace`, `@Code`, `@DB_Expect`, `@Log_Expect`.
2. **Шаг Б**: В `QAFixtureFactory` (`fixtures_data.py`) пишется асинхронный метод генерации изолированного SQL-кадра.
3. **Шаг В**: Пишется Python-скрипт шагов (`_steps.py`), вызывающий фабрику, совершающий HTTP-запросы и жесткие СУБД/Логгер ассерты.
4. **Шаг Г**: Код роутеров бэкенда пишется *после* создания теста, до тех пор, пока СУБД-ассерт не станет зеленым.
